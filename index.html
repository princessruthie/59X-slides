Now

  1. check to see if you have git by typing: git status
  2. if command not found, download it https://git-scm.com/downloads

Staging

  • git uses a sort of two-step. You have to
    1. add the files
    2. commit
  • When you add a file, it's now staged (in your cart). Then you can commit(buy).
  • Any changes that aren't added before the commit are kinda left in your 'wishlist'.
  • This metaphor isn't perfect. (mostly because there's a notion of 'checkout')

Quick, dirty

  • git clone [wherever the repo is]
  • git log
  • git status
  • ~~change a file and save~~
  • git add [filename or a period]
  • git commit -m "feat: Implement BFS for karel" **avoids using vim
  • git push
  • git reset [filename or just a period]

Undoing stuff

  3. to get the id, run git log

Ignorance is bliss

Not everything should go to the cloud. EVEN in private repos. Use a .ignore file to keep these files safe/local.

  • passwords
  • keys
  • IDE config files
  • lets do our own and then
  • check out

Things to know

  • git tracks CHANGES. Can think like a bank account where to undo, you just change signs and rerun transactions in reverse order to get to a previous balance.
  • Hacktoberfest
  • Expressive commit histories
  • Even though git looks like a log, everything can be faked or overwritten

Do we have time to `git` into branches?

  • You can do everything on one branch. But you shouldn't.
  • You can instead checkout your a branch, do several commits untiil you have a nice feature, then merge it back
  • Keep the master branch only for "ready" code. Whatever that means to you.
  • On GH, if you have a branch called gh-pages, GH will serve the files like a website

Now

  1. git checkout -b name-of-new-branch
  2. git branch (this tells you the names of all your branches)
  3. git checkout master (no -b because master already exists)
